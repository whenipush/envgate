# EnvGate

Система централизованного управления переменными окружения и конфигурациями распределенных приложений. Позволяет хранить параметры сред в едином контуре и динамически доставлять их на конечные хосты через системный агент.

## 1. Назначение системы

EnvGate решает задачу безопасного хранения и распределения конфигураций (Environment Variables) по разным контурам разработки (`dev`, `stage`, `prod`). 

**Основные функции:**
* Разделение конфигураций в рамках изолированных проектов и сред.
* Изоляция секретов: конечные приложения не хранят `.env` файлы на диске, а получают параметры напрямую в оперативную память.
* Контроль доступа: генерация уникальных токенов для агентов с привязкой к конкретному проекту, окружению и назначению.

## 2. Архитектура и Стек технологий

Проект реализован в виде монорепозитория, разделенного на два основных компонента: сервер управления (серверная часть + веб-панель) и клиентский системный агент (CLI).

* **Backend:** Go (фреймворк Gin для веб-панели и REST API).
* **Database:** Embedded Key-Value хранилище на базе `bbolt` (BoltDB). Данные сохраняются в один локальный файл, установка внешней БД не требуется.
* **Frontend / UI:** Go-templates, интерактивность через **HTMX** (использование Morph-вставок без полной перезагрузки страниц), стилизация — Tailwind CSS.
* **Связь Сервер-Агент:** gRPC / Protocol Buffers (протокол HTTP/2).

## 3. Конфигурация сервера

Перед запуском сервера (через Docker или локально) необходимо создать файл конфигурации окружения `.env` в корневой директории проекта.

Пример конфигурационного файла (`.env.server.example`):

```env
MASTER_KEY=secret         # Ключ для шифрования токенов/секретов в БД
GRPC_PORT=50051           # Порт gRPC-сервера для подключения агентов
REST_PORT=3000            # Порт HTTP REST API и панели администрирования
DB_PATH=envgate.db        # Путь к файлу базы данных bbolt
AUTH_USERNAME=admin       # Логин для входа в веб-панель
AUTH_PASSWORD=admin       # Пароль для входа в веб-панель
```

## 4. Развертывание сервера

### Вариант А: Docker и Docker Compose (Рекомендуемый)

Для развертывания в изолированном контейнере используются имеющиеся в репозитории `Dockerfile` и `docker-compose.yml`.

1. Запуск всего окружения в фоновом режиме:

```bash
docker compose up -d --build
```

2. Остановка контейнеров:

```bash
docker compose down
```

Панель администрирования станет доступна в браузере по адресу `http://localhost:3000` (или согласно `REST_PORT` из `.env`).

### Вариант Б: Локальный запуск (Сборка из исходников)

**Требования:** Установленный Go версии 1.20 или выше.

1. Клонирование репозитория и переход в директорию проекта:

```bash
git clone https://github.com/whenipush/envgate.git
cd envgate
```

2. Запуск сервера администрирования напрямую через Go toolchain:

```bash
go run cmd/server/main.go
```

Файл базы данных `envgate.db` (согласно `DB_PATH`) автоматически создастся в указанном каталоге при первом старте.

## 5. Панель управления (UI)

После авторизации в веб-интерфейсе доступны следующие модули:

* **Управление переменными:** Вкладки мгновенного переключения сред (например, `development`, `production`). Интерфейс добавления, изменения и удаления пар `KEY=value` работает на базе HTMX без перезагрузки страниц.
* **Выпуск токенов (Ключи доступа):** Форма генерации ключей для агента. При создании указывается целевое окружение и текстовое описание назначения (например, `k8s-backend-pod` или имя разработчика). Токен отображается один раз с возможностью быстрого копирования в буфер обмена.

## 6. Системный агент (`envgate-agent`)

Сборка и установка бинарного файла агента в систему может быть выполнена двумя способами.

### Сборка и установка

#### Способ 1: Установка из удаленного репозитория (Релизная сборка)
Для установки агента на целевой сервер без необходимости клонировать весь монорепозиторий целиком:
```bash
go install [github.com/whenipush/envgate/cmd/envgate-agent@latest](https://github.com/whenipush/envgate/cmd/envgate-agent@latest)
```

#### Способ 2: Установка из локальных исходников (Для разработки)

Для сборки агента из текущих локальных исходников проекта (запускать из корня репозитория):

```bash
go install ./cmd/envgate-agent
```

### Конфигурация окружения Агента

Перед запуском агента необходимо объявить следующие переменные:

| Переменная | Обязательное | Дефолтное значение | Описание |
| --- | --- | --- | --- |
| `ENVGATE_TOKEN` | **Да** | *отсутствует* | Токен доступа к проекту/среде, сгенерированный в веб-панели. |
| `ENVGATE_SERVER_ADDR` | Нет | `localhost:50051` | Адрес gRPC-сервера EnvGate (хост и порт). |
| `ENVGATE_SERVER_INSECURE` | Нет | `true` | Использовать незашифрованное gRPC-соединение (полезно для локальной разработки). |

---

### Способы запуска и использования

#### Вариант 1: Одноразовый запуск с передачей ENV в строке (Рекомендуемый для тестов)


Если сервер EnvGate запущен на удаленном хосте:

```bash
ENVGATE_SERVER_ADDR="192.168.1.100:50051" ENVGATE_TOKEN="eg_tok_your_secret_token" envgate-agent ./your-application-binary
```

#### Вариант 2: Экспорт переменных на всю сессию терминала

Если вы постоянно перезапускаете приложение в процессе разработки, можно экспортировать переменные один раз для текущей вкладки терминала:

```bash
export ENVGATE_TOKEN="eg_tok_your_secret_token"
export ENVGATE_SERVER_ADDR="localhost:50051"
```
```
envgate-agent ./your-application-binary
```
